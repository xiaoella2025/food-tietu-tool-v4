当前最重要的是第三步：文字上图。

现在第三步页面已经有基础框架：

- 中间：画布预览
- 右侧：文案内容

- 底部：配方相关按钮

## 七、第三步「文字上图」的目标

第三步先做成最小可用版本，不要做复杂设计器。

最小可用目标：

1. 左侧选择一张待加字图片
2. 右侧输入标题 / 步骤 / 正文
3. 点击「加入画布」
4. 文字出现在当前图片上
5. 文字可以选中
6. 文字可以拖动
7. 文字图层列表能显示当前图层
8. 选中图层后可以修改字号、颜色、描边等基础样式
9. 可以删除图层
10. 可以保存当前文字布局为配方
11. 下拉框能看到已保存配方
12. 必须先选择配方，再点击应用配方
13. 应用配方只应用到当前图片
14. 保存加字成品后，生成一张成品图
15. 成品图可以被第四步读取

第一轮真正开发时，优先做前 8 项。

配方和成品保存可以作为第二小步，但不要一次性大改。

---

## 八、第三步配方逻辑要求

配方不是最终图片。

配方应该保存的是文字模板信息，例如：

- 图层类型：标题 / 步骤 / 正文
- 图层文字内容或占位类型
- x / y 位置
- 字号
- 字体
- 字体颜色
- 描边颜色
- 描边粗细
- 背景板设置
- 对齐方式
- 图层顺序

配方不应该绑定死：

- 某一张图片
- 某一个素材编号
- 某一次导出的 dataUrl
- 某个固定图片尺寸

应用配方时，应基于当前画布尺寸按比例恢复位置。

如果当前画布尺寸不同，应做简单比例换算。

---

## 九、第三步禁止出现的问题

请重点避免：

1. 保存了配方，但下拉框里看不到
2. 点「应用配方」时还没选择配方就自动乱套
3. 应用配方后文字位置错乱
4. 加文字后图层列表不更新
5. 删除图层后画布文字还在
6. 切换图片后图层串到别的图片
7. 保存成品后找不到成品图
8. 修改第三步时影响第二步图片编辑
9. 为了修文字上图而重写全局 state
10. 为了修 UI 而重构整个 app.js

## 十三、成本控制要求

本项目第一阶段不要增加使用成本。

不要接入：

- AI 文案 API
- AI 生图 API
- AI 高清增强 API
- 云端数据库
- 云端存储
- 登录系统
- 复杂授权系统
- 飞书同步
- 自动发布

如需要 AI 文案或生图，低成本方案是：

工具内置提示词 → 助理复制到豆包/即梦 → 外部生成 → 再上传或粘贴回工具。
